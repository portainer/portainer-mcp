# Docker / Kubernetes proxy tools

The proxy tools expose Portainer's `/endpoints/{id}/docker/...` and
`/endpoints/{id}/kubernetes/...` path-forwarders as MCP tools. These paths
aren't fully enumerated in the OpenAPI spec (they forward arbitrary
subpaths), so they're hand-written in `src/portainer_mcp/proxy.py` rather
than generated by `FastMCP.from_openapi`.

## What's shipped (Option 2: server-side projection)

Two symmetric tools, each accepting an optional JMESPath `select`
expression applied to the response **before** it reaches the model:

| Tool | Methods | `select` |
|------|---------|----------|
| `docker_proxy` | any | yes |
| `kubernetes_proxy` | any | yes |

A hard char cap (default `75_000`) truncates oversized responses with a
hint telling the model to narrow `select`. Override with
`PORTAINER_PROXY_MAX_CHARS=<int>`. The cap is char-based by design: it's
a safety valve, not a precision token meter. Dense JSON (Docker/K8s
payloads full of IDs, hashes, nested structure) packs at ~3 chars/token
rather than the prose-text ~4, so 75k chars targets ~25k tokens with
margin. A proper tokenizer (tiktoken / Anthropic `count_tokens`) would
narrow the variance from ~30-50% to ~15-25%, but at the cost of a
dependency and per-call overhead that isn't worth it for a truncation
guard.
`PORTAINER_READ_ONLY=1` rejects non-GET calls at tool-invocation time.

### Why this design

- **Filter server-side, not in the model.** A `select` of
  `[].{id:Id,name:Names[0],state:State}` against `/containers/json` for
  50 containers ships ~500 tokens instead of ~10k.
- **No per-platform special-casing.** Docker and Kubernetes are
  symmetric. We deliberately don't ship a K8s `managedFields` stripper:
  the model's `select` projection drops it for free, and stripping one
  arbitrary field on one platform while ignoring equally noisy Docker
  fields doesn't scale.
- **One escape hatch.** The same tool covers writes — no separate raw
  variant. The `method` parameter takes any HTTP verb.

## What to measure during testing

Track these to decide if Option 2 is enough:

1. **Hit rate of `select`.** What fraction of proxy calls supply one?
   If low, the model needs better tool descriptions or examples in the
   system prompt — not a different design.
2. **Truncation rate.** How often does the 80k cap fire? If frequent on
   filtered responses, the data is genuinely large per-object and
   filtering alone won't fix it.
3. **Round trips per workflow.** Count Portainer hits per resolved
   question. Two hits (broad → narrow) is acceptable; four+ suggests the
   model is groping and would benefit from a cached scratchpad.
4. **JMESPath errors.** A high rate signals the model struggles with the
   syntax and we should switch to `jq` or simplify guidance.

## Cheap fallback if K8s reads are chronically unfiltered

If testing shows the model omits `select` on K8s reads often enough that
raw responses dominate context, the lightest intervention is to strip
`metadata.managedFields` unconditionally on every K8s response (before
`select` runs). Roughly 20 lines, no new tool, no schema change. Only
reach for this if metrics demand it — otherwise it's special-casing in
search of a problem.

## Planned evolution (Option 3: resource spillover)

If truncation fires often *and* the second-hit rate is high, layer a
cache + MCP resource on top of the existing proxy:

```
docker_proxy/kubernetes_proxy  →  full response size > budget?
                                    ├── no  → return as today
                                    └── yes → cache under stable key
                                              return {preview, uri, index}
                                              + companion tool:
                                              query_cached_response(uri, jq)
```

Sketch:

- LRU cache keyed by `(env_id, method, path, sorted query_params)`, TTL
  ~10 min, byte cap ~50 MB. `cachetools.TTLCache` is enough.
- Register dynamic `@mcp.resource("portainer://proxy/{uuid}")` that
  returns the cached body.
- Preview = first ~2k chars + top-level JSON keys/item count.
- Companion `query_cached_response(uri, jq_or_jmespath)` tool so the
  model can extract narrower views *without re-hitting Portainer*.

Pulling this in is purely additive — the existing tools keep their
shape; the spillover branch is an extra return path. Don't build it
until the metrics above say it's needed.

## Things deliberately not done

- **Streaming / progress.** Long Docker logs and exec sessions are
  better served by dedicated tools (`docker_logs`, `docker_exec`) once
  we know they're worth the surface area.
- **Auto-discovery of API shape.** Tempting to inject a schema
  description per call, but JMESPath is more powerful and the model
  already knows Docker/K8s shapes well enough.
- **Multiple filter languages.** JMESPath only for now; switch to `jq`
  wholesale if testing shows JMESPath is the blocker.
